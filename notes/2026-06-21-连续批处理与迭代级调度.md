# 摘要

文章梳理了 LLM 在线推理调度从 Static Batching、Dynamic Batching 到 Continuous Batching 的演进。核心变化是把调度粒度从“整个请求”缩小到“每次模型迭代”：完成的请求可立即退出，等待请求可立即补入，从而减少队头阻塞和批次空洞。

在此基础上，文章继续介绍 Orca 的迭代级调度、vLLM 的 PagedAttention、Chunked Prefill、Sarathi-Serve 的混合调度、Prefill-Decode 分离，以及通过减少或隐藏 CPU 调度开销来提高 GPU 利用率的方案。全文的主线是：推理服务优化不能只看单次模型计算速度，还要联合考虑请求调度、KV Cache、工作负载形态和服务级目标。

原始网页已保存为 `raw/2026-06-21-cnblogs-19964639.html`。正文标注“技术日报 2026-04-21”，博客页面实际发布于 2026-05-01。

# 核心观点

1. LLM 推理包含 Prefill 和 Decode 两个特性不同的阶段。Prefill 可对输入 token 并行计算，偏计算密集；Decode 自回归逐 token 生成，更容易受访存、KV Cache 和调度开销限制。
2. Static Batching 在批次建立后一直等待最长请求，短请求完成后留下计算空洞；Dynamic Batching 改善了组批时机，但没有消除批次内部的队头阻塞。
3. Continuous Batching 在每次迭代后重新决定批次成员。请求一旦结束即可释放资源，新请求则可进入下一次迭代，这是 Iteration-Level Scheduling 的关键。
4. 连续批处理需要更灵活的 KV Cache 管理。文章以 vLLM 的 PagedAttention 为例，说明把 KV Cache 切成固定大小的块，可以降低连续内存预留、碎片和扩容复制带来的浪费。
5. 长 Prefill 会阻塞正在 Decode 的请求，造成 token 间延迟升高。Chunked Prefill 把长输入切成多个小块，并在块之间穿插 Decode，以吞吐损失换取更稳定的交互延迟。
6. Sarathi-Serve 进一步把 Prefill chunk 与 Decode token 混合成较均衡的批次，目标是减少不同迭代间的计算量波动。
7. Prefill-Decode 分离让两个阶段使用独立实例和硬件配置，以便分别优化 TTFT 与 TBT；代价是增加 KV Cache 传输、路由和集群管理复杂度。
8. 当 GPU 计算越来越快时，CPU 调度和同步可能成为瓶颈。SGLang 零开销调度和 vLLM Multi-Step 调度分别通过重叠调度与计算、一次安排多个 Decode step 来降低这部分开销。
9. 调度策略没有脱离工作负载的统一最优解。请求长度分布、并发量、TTFT/TBT SLO、显存预算和硬件成本共同决定参数与架构选择。

# 我的理解

连续批处理的本质不是简单扩大 batch，而是把 batch 从固定容器变成随迭代变化的资源集合。它提高了资源利用率，但也把问题从“如何组一个批次”升级为“如何持续决定谁先运行、运行多少 token、占用多少 KV Cache”。

优化目标也不是单一吞吐量。在线交互场景至少要同时观察：

- TTFT：用户等待第一个 token 的时间，容易受排队和 Prefill 影响。
- TBT：相邻输出 token 的间隔，容易受长 Prefill 插入和调度抖动影响。
- TPS 或总体吞吐：单位时间处理的 token 或请求数量。
- KV Cache 利用率：决定可容纳的并发序列和上下文规模。

文章列出的性能提升、告警阈值和具体参数值是作者给出的经验或引用结果，未在本知识库环境中独立复测。尤其是 `max_num_batched_tokens`、`num_scheduler_steps` 等参数会随 vLLM 版本、模型、GPU 和流量特征变化，使用前应回查当前版本文档并压测。

# 可复用方法

1. 先描述工作负载：统计 prompt 长度、输出长度、并发请求数和请求到达分布，不直接套用固定参数。
2. 同时记录 TTFT、TBT、吞吐和 KV Cache 利用率，并至少比较 P50 与 P99，避免平均值掩盖长尾。
3. 以基线配置运行压测，再单独调整 `max_num_batched_tokens`、最大并发序列数、显存利用率和调度步数，记录每次变更的收益与代价。
4. 长 prompt 导致 TBT 波动时，优先验证 Chunked Prefill；若 Prefill 与 Decode 的资源需求长期冲突，再评估阶段分离。
5. 高并发下 KV Cache 成为限制时，检查分页管理、缓存精度、前缀复用和并行策略，而不是只增加 batch size。
6. 生产选型时先确定 SLO，再在满足 SLO 的配置中比较吞吐和成本。

# 项目关联

这篇资料补充了 `wiki/大模型部署.md` 中关于 vLLM 和在线推理服务的系统层知识，可用于设计个人知识库问答服务的部署实验。若后续从单用户原型扩展到多用户服务，应把 TTFT、TBT、KV Cache 占用和长上下文干扰纳入验收指标。

对当前个人知识库而言，优先级仍应是跑通可复现的模型服务和问答链路；Continuous Batching、Chunked Prefill 与 Prefill-Decode 分离更适合作为并发和性能问题出现后的优化手段。
