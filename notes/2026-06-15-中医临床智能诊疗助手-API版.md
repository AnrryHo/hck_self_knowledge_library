# 摘要

这份课件是中医临床智能诊疗助手的 API 实现版本，使用 DashScope 的语言模型和嵌入模型，讲解语料读取、切分、向量索引、查询引擎、索引持久化、检索追踪、Prompt 定制，并延伸到 RAG 评估和 WebUI 开发。

# 核心观点

1. API 方案省去了本地模型权重部署，但需要配置凭据、模型参数并管理调用成本和外部依赖。
2. LlamaIndex 的全局 LLM 与 Embedding 配置决定后续索引和查询所使用的模型。
3. 文档切分、嵌入、索引、检索和答案生成仍是核心流程，与本地模型版的架构基本一致。
4. 课件加入 RAG 系统评估和 WebUI，展示从 Notebook 原型向可交互应用扩展的方向。
5. 医疗问答输出需要证据追踪和领域验证，模型 API 并不会自动保证专业正确性。

# 我的理解

API 版与本地模型版的主要差异在模型供应和运行边界，而不是 RAG 的基本结构。比较两份课件有助于理解部署成本、隐私、可控性和开发速度之间的权衡。

# 可复用方法

- 通过环境变量管理 API Key，不将凭据写入 Notebook 或知识库。
- 在固定评测集上比较不同模型配置的召回和回答效果。
- 持久化向量索引，避免重复计算和不必要的 API 成本。
- WebUI 只负责交互，检索证据和错误信息仍需可观察。
- 医疗应用增加免责声明、来源展示、人工复核和敏感数据保护。

# 项目关联

这份资料适合作为 API 型 RAG 项目模板，并可与本地模型版对照学习部署方案选择。

# 需要复习的知识点

- API 模型与本地模型在成本、隐私和运维方面的差异。
- LlamaIndex 如何配置全局 LLM 和 Embedding。
- 为什么向量索引需要持久化。
- RAG 评估应覆盖哪些环节。
- WebUI 如何保留检索证据和错误可观察性。
