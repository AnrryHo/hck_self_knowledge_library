# 摘要

这篇文章把 Codex 的斜杠命令和快捷键视为“会话控制台”：它们不是给模型派活，而是控制 Codex 程序本身，例如切模型、查状态、清上下文、审 diff、进入计划模式。CLI、桌面 App、IDE 扩展的命令集合不同，最可靠的方式是在当前入口输入 `/` 查看真实菜单。

# 核心观点

1. **斜杠命令是元操作**：它面向 Codex 程序本身，负责模型、状态、权限、上下文和审查等控制，不等同于普通自然语言任务。
2. **入口不同，命令不同**：CLI 命令最多，桌面 App 保留少量高频命令，IDE 扩展增加 `/local`、`/cloud`、`/auto-context` 等编辑器相关能力。
3. **CLI 命令应按流程记忆**：搭场子用 `/init`、`/permissions`、`/status`；干活中用 `/model`、`/compact`、`/clear`、`/new`、`/plan`；交付前用 `/diff`、`/review`。
4. **容易混淆的三个命令**：`/clear` 是清屏并换新上下文，`/new` 是不清屏换新对话，`/compact` 是压缩当前上下文后继续。
5. **快捷键覆盖高频动作**：`Ctrl+L` 只清屏不丢上下文，`Ctrl+O` 复制最近完成输出，`Tab` 可排队输入，`Ctrl+G` 可用外部编辑器写长提示，行首 `!` 可运行 shell 命令并把输出纳入上下文。

# 我的理解

斜杠命令的价值在于把“控制 Codex”与“让 Codex 做任务”分离。遇到会话管理问题时，不应该重新启动或用自然语言绕弯，而应该优先想到是否有现成的 `/` 命令。

推断：对本知识库维护来说，最常用的组合会是 `/status` 确认上下文和权限、`/plan` 处理批量入库或重构前的方案、`/diff` 检查未跟踪文件、`/review` 提交前审查，以及 `!python scripts/fuxi.py validate` 这类本地检查命令。

# 可复用方法

- **进入新仓库**：先用 `/status` 看当前模型、权限和上下文；必要时 `/init` 或阅读已有 `AGENTS.md`。
- **大任务开始前**：用 `/plan` 先生成方案，再决定是否执行。
- **上下文管理**：同一任务过长用 `/compact`，换无关任务用 `/clear` 或 `/new`。
- **提交前检查**：用 `/diff` 看所有改动（含未跟踪文件），必要时 `/review` 做行为和测试风险审查。
- **高频命令**：用 `!git status`、`!python scripts/fuxi.py validate` 快速运行检查，并让 Codex 看到输出。

# 项目关联

- 本项目要求入库后更新索引、关联和主题页，`/diff` 可帮助确认新增 raw、notes、wiki、INDEX 是否完整。
- `Ctrl+L` 适合长时间整理资料时清屏但保留上下文，避免误用 `/clear` 丢失当前任务背景。
- IDE 的 `/local` 与 `/cloud` 可用于在本地资料整理和云端批量处理之间切换，但具体可用性要以当前 IDE 扩展菜单为准。
