# 摘要

原文链接：https://blog.csdn.net/2401_84059420/article/details/158385498

这篇文章记录了在 Ubuntu 24.04.3 服务器上部署 Qwen-7B-Chat 大模型的实践流程：先用 conda 创建运行环境并安装 PyTorch/CUDA 依赖，再用 git-lfs 下载模型，随后用 transformers 加载模型，封装 FastAPI 接口，最后用 Streamlit 做一个隧道工程问答 Web 前端。文章目标不是通用模型平台，而是实现一个可访问的“隧道工程专业问答系统”。

# 核心观点

1. 服务器部署大模型要先解决 Python 环境、PyTorch、CUDA、NumPy 等基础依赖问题，原文使用 `conda create -n ai_dev` 和 `conda install pytorch==2.2.0 ... pytorch-cuda=12.1`。
2. 环境配置完成后，需要用脚本检查 PyTorch 版本和 `torch.cuda.is_available()`，确认 GPU 能被识别。
3. 原文通过 `git-lfs` 从 ModelScope 下载 `Qwen-7B-Chat` 到服务器目录，例如 `/home/ubuntu/models/llm/Qwen-7B-Chat`。
4. 模型加载阶段用 transformers 的 `AutoTokenizer`、`AutoModelForCausalLM` 和 `GenerationConfig`，并显式指定 `cuda:0`、`torch.float16` 等运行配置。
5. 作者先写了本地推理脚本 `llm_deploy.py`，再进一步封装为 `tunnel_llm_api.py`，用 FastAPI 提供 `/api/chat` 接口。
6. API 设计支持多种任务类型：通用问答、意图识别、预警生成、病害预测、碰撞解析；其中结构化任务通过提示词要求模型输出 JSON。
7. 服务运行使用 `CUDA_VISIBLE_DEVICES=0 nohup python tunnel_llm_api.py > llm_api.log 2>&1 &`，并通过 `tail -f llm_api.log` 观察加载日志。
8. 前端使用 Streamlit 快速搭建聊天页面，通过 HTTP 请求调用 FastAPI 服务，最后用 `streamlit run ... --server.address 0.0.0.0 --server.port 8501` 对内网开放。

# 我的理解

这篇文章比上一篇“本地部署入门”更接近实际项目落地：它不只是让模型能对话，而是把模型部署成一个后端 API，并提供一个简单 Web 入口。它体现了一个最小可用链路：

```text
服务器环境
→ 模型权重
→ 本地推理脚本
→ FastAPI 服务
→ curl 测试
→ Streamlit 前端
→ 后台常驻运行
```

推断：这套流程适合个人或小团队做原型验证，但还不等于生产级部署。原文没有系统处理鉴权、限流、并发、队列、异常恢复、日志轮转、服务守护、容器化、HTTPS、模型热更新、提示词注入防护和数据安全等问题。

# 可复用方法

- 环境先行：创建独立 conda 环境，安装 PyTorch/CUDA 后立即运行 `torch.cuda.is_available()` 做可用性检查。
- 依赖冲突处理：遇到 NumPy、transformers、tiktoken、einops、auto-gptq、optimum 等依赖问题时逐项补齐，并记录最终可运行版本。
- 模型目录固定：把项目代码和模型权重分开放置，例如 `~/ai_project` 存代码，`~/models/llm` 存 LLM 权重。
- 先本地脚本，后服务化：先用单文件推理脚本验证模型能加载和生成，再封装 FastAPI，降低排错复杂度。
- API 任务分流：用请求字段 `type` 区分通用问答和结构化任务，后端根据任务类型选择不同提示词和生成参数。
- 后台运行：用 `nohup`、日志文件和 `tail -f` 做最基础的后台服务观察。
- 前端快速验证：用 Streamlit 直接调用 API，比手写完整前端更适合原型阶段。
