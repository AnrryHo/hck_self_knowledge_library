# 大模型部署

大模型部署是把模型从“可下载的权重或工具”变成“本机或服务中可调用能力”的过程。当前库内资料主要覆盖本地入门部署、RAG 项目中的本地/API 模型接入，以及面向检索问答的应用集成。

## 本地部署路径

1. 应用部署：使用 Ollama、LM Studio 等工具，适合先验证模型能否在本机运行。
2. 源码部署：使用 transformers、vLLM、llama.cpp 等路线，需要自行处理 Python 环境、依赖、模型权重、CUDA/显卡和启动代码。
3. API 接入：不在本机运行模型，通过外部模型服务完成推理，工程入口更快，但要考虑成本、网络、数据边界和服务依赖。

## 当前结论

- 初学者先用 LM Studio 或 Ollama 跑通小模型，比直接源码部署更稳。
- 本地部署前要先匹配模型大小和硬件资源，尤其是内存、显存和量化版本。
- 源码部署的主要风险通常不在模型调用代码，而在 Python、PyTorch、CUDA、驱动和模型权重版本匹配。
- 本地模型适合隐私、自主控制和离线实验；API 模型适合快速验证和减少本地运维。
- 若目标是知识库问答，部署只是底座，还需要文档加载、切分、检索、上下文组装和评估。

## 在线推理调度

- LLM 推理包含 Prefill 与 Decode 两个计算特性不同的阶段，在线服务需要同时优化首 token 延迟、token 间延迟、吞吐和 KV Cache 占用。
- Continuous Batching 把调度粒度缩小到每次模型迭代，使完成的请求及时退出、新请求及时补入，减少固定批次中的队头阻塞和计算空洞。
- PagedAttention 通过分页式 KV Cache 管理支持更灵活的请求进入、退出和扩展。
- Chunked Prefill 把长输入切块并与 Decode 交错执行，以部分吞吐代价换取更稳定的交互延迟。
- Prefill-Decode 分离可以分别优化两阶段的硬件与调度，但会增加 KV Cache 传输和集群管理复杂度。
- 具体调度参数依赖模型、推理引擎版本、GPU 与请求长度分布，应以真实工作负载压测为准。
