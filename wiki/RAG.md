# RAG

RAG（检索增强生成）通过从外部知识源检索相关内容，为大语言模型生成答案提供上下文。当前知识库中的课程覆盖了它的基础设施、框架实现、召回优化、评估和项目落地。

## 核心链路

```text
原始文档
→ 加载与清洗
→ 文本切分或结构化解析
→ Embedding
→ 向量索引
→ 查询改写与检索
→ 重排序或后处理
→ 上下文组装
→ LLM 生成
→ 质量评估
```

## 共同结论

1. Embedding 和向量数据库构成基础检索能力，但相似性检索本身不等于高质量 RAG。
2. 文档切分、Top-K、查询方式、重排序和响应合成都会影响最终答案。
3. LangChain 和 LlamaIndex 都能构建 RAG；前者侧重应用组件与编排，后者在数据接入、索引和检索抽象方面提供了更完整的教学案例。
4. 调试时需要保留命中文档、检索分数、最终上下文和模型答案，才能判断问题发生在哪个阶段。
5. RAG 优化需要固定评测集和可重复指标，不能只凭单次问答体验判断。

## 差异与权衡

- **本地模型与 API 模型**：本地模型更强调隐私和自主控制，但增加模型下载、硬件和运维负担；API 模型接入更快，但带来调用成本、外部依赖和数据边界问题。
- **LangChain 与 LlamaIndex**：课程内容显示两者能力存在重叠。当前资料更适合把 LangChain 理解为通用应用编排工具，把 LlamaIndex 理解为知识接入与检索框架；这不是严格的产品边界。
- **基础检索与高级检索**：基础向量 Top-K 适合建立原型，高级方案应由具体失败案例驱动，不应一次性堆叠所有技术。

## 当前判断

对个人知识库而言，实施顺序应是：

1. 先建立可追踪的基础检索链路。
2. 准备真实问题和参考资料作为评测集。
3. 根据失败类型调整切分、Top-K 和元数据。
4. 仍有明确问题时再加入查询改写、融合检索或重排序。
5. 最后评估是否值得引入复杂框架和独立向量数据库。

这是基于当前六份课程资料做出的综合判断。知识库尚未实际运行 RAG 实验，因此没有库内数据能够证明某种框架或优化方法效果最好。

## 后续研究

- 为本知识库建立小规模问答评测集。
- 比较关键词检索与向量检索的实际增益。
- 验证不同切分策略对中文长文的影响。
- 建立检索命中、答案依据和失败类型的记录格式。
- 对 Ragas 指标与人工判断的一致性进行实验。
- 明确医疗等高风险领域的证据、隐私和人工复核要求。
